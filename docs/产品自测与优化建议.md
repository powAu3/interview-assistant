# 面试流程 · 产品自测与优化建议

> 以产品视角走查「开始 → 录音/转写 → 问答 → 取消/清空」全流程，列出可优化点。

---

## 一、流程自测结论（当前实现）

| 环节 | 状态 | 说明 |
|------|------|------|
| 进入页面 | ✅ | 拉 config/devices/options，失败时全屏「连接后端失败」+ 重试 |
| 设备选择 | ✅ | 有系统音频/麦克风分组，无 loopback 时有提示，选麦克风时建议用系统音频 |
| 开始面试 | ✅ | 需先选设备，有 loading，失败用 error 条展示 |
| 暂停/继续 | ✅ | 有 loading，状态条（录制中/已暂停）清晰 |
| 实时转写 | ✅ | 转写中 + 音量条，空状态「等待语音输入」/「点击开始面试」 |
| 自动出题 | ✅ | 语音识别到问题后自动入队生成答案 |
| 手动提问 | ✅ | 输入框 + 发送，支持粘贴截图（当前模型不支持时有提示） |
| 答案流式 | ✅ | 思考过程可折叠，生成中 + 光标，支持取消生成 |
| 取消生成 | ✅ | 控制栏出现「取消生成」，WS 收 answer_cancelled 后该条显示 [已取消] |
| 清空会话 | ⚠️ | 见下文优化项 |
| 简历上传 | ⚠️ | 见下文优化项 |
| 后端异常 | ⚠️ | 见下文优化项 |

## 二、建议优化项（按优先级）

### 1. 清空会话：防误触 + 反馈（高）

- **现象**：点击「清空」立即调用 `api.clear()` 并 `clearSession()`，无二次确认、无 loading、无成功提示。
- **风险**：误点会清空所有转录和 QA，且不可恢复。
- **建议**：
  - 有内容时弹确认：「确定要清空当前会话吗？转录与答案将全部清除。」
  - 清空时加短暂 loading 或至少按钮 disabled，避免连点。
  - 可选：清空后 toaster「已清空」或 1s 内提示，让用户确认操作生效。

### 2. 后端/面试异常：用户可见（高）

- **现象**：WS 收到 `type: 'error'` 时仅 `console.error('[WS]', msg.message)`，界面无提示。
- **影响**：面试循环异常、转写失败、模型错误等，用户不知道原因。
- **建议**：
  - 在 store 中增加 `lastWSError: string | null`，WS 收到 error 时 set 并 3～5s 后自动清空。
  - 在 ControlBar 上方或 Header 旁展示该错误（与现有 error 条样式一致），可点击关闭。

### 3. 取消生成：操作反馈（中）

- **现象**：点击「取消生成」只调 `api.cancelAsk()`，无 loading、无 toast。
- **影响**：网络稍慢时用户不知道是否已发送，可能多次点击。
- **建议**：点击后按钮短暂显示「取消中」或 loading，或请求成功后 toaster「已发送取消」；若后端已支持快速响应，可仅加 200ms 内的防抖/loading 状态。

### 4. 「已取消」答案的展示（中）

- **现象**：取消后该条答案为 `[已取消]`，与正常 Markdown 一起渲染。
- **建议**：当 `qa.answer === '[已取消]'` 时单独展示样式（如灰色斜体 + 小图标），不按 Markdown 渲染，避免与正常内容混淆。

### 5. 手动提问失败：保留输入内容（中）

- **现象**：`handleAsk` 里先 `await api.ask(...)`，成功后才 `setManualQuestion('')`；若 ask 失败会 `setError(e.message)`，但若在清空输入后再失败（或接口返回慢），用户可能已丢失输入。
- **现状**：当前是成功后才清空，逻辑正确；若将来改为「先清空再请求」需避免失败后内容丢失。
- **建议**：保持「请求成功后再清空」；若增加乐观清空，失败时需恢复原内容或 toaster 提示「发送失败，请重试」。

### 6. 简历上传：上传中状态（低）

- **现象**：上传 PDF/DOC 时无 loading，大文件或 VL 解析慢时用户易以为无响应。
- **建议**：上传时按钮或区域显示 loading（如「解析中…」），收到响应后再关闭；解析失败时已有 setError，保持不变即可。

### 7. 首次使用引导（低）

- **现象**：答案区空状态有「AI 答案将在这里显示」和说明，但未强调需先「开始面试」或选设备。
- **建议**：在无转录、无 QA 时，答案区或控制栏上方可加一句「请先选择音频设备并点击「开始」开始面试」，减少冷启动困惑。

### 8. 清空时若正在生成（低）

- **现象**：`api.clear()` 会 abort 并清空队列，前端 `clearSession()` 会清掉所有 QA；若当前有一条正在生成，会先收到 answer_cancelled 再清空，顺序上可能有一瞬「已取消」再消失。
- **建议**：可接受；若希望更顺滑，可在 clear 时前端先立刻 clearSession，再调 api.clear()，避免短暂出现「已取消」再清空（与后端 abort 无冲突）。

---

## 三、小结

- **核心流程**：开始 → 录音/转写 → 自动或手动出题 → 流式答案 → 取消/清空，链路通畅，状态与错误在多数环节有处理。
- **优先建议**：清空防误触与反馈、WS 服务端错误对用户可见、取消生成的轻量反馈；其次为「已取消」样式、简历上传 loading、首用引导。
- 以上均为体验与可感知性优化，不影响现有接口与业务逻辑，可按迭代节奏分批做。
